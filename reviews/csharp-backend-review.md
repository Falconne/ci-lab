**File:** `src/be/Mergician/Services/AutoMerge/AutoMergeService.cs:115-127`

## Mergician — Services

### 8. `UserActivityBackgroundSyncService` polling loop exits on recovery mode instead of waiting

**File:** `src/be/Mergician/Services/UserActivityBackgroundSyncService.cs:168-175`

The project guideline says background services must handle GitLab going down by retrying indefinitely. `AutoMergeService` conforms to this by calling `WaitForReady` when recovery mode is signalled. `UserActivityBackgroundSyncService` instead breaks out of the polling loop and exits the thread entirely when `IsInGitLabRecoveryMode` is true, relying on the user making new HTTP requests to restart the thread.

This is inconsistent and error-prone: if no requests arrive during recovery (e.g. a browser tab is open but idle), the sync thread is never restarted. The thread should wait for recovery mode to clear rather than exiting.

```csharp
// Current — thread exits
if (_gitLabRecoveryService.IsInGitLabRecoveryMode)
{
    _logger.LogInformation("Stopping sync thread for user {UserId}: GitLab recovery mode is active", gitLabUserId);
    break;
}

// Fix — wait for recovery to clear before continuing, mirroring AutoMergeService
if (_gitLabRecoveryService.IsInGitLabRecoveryMode)
{
    _logger.LogInformation(
        "Sync thread for user {UserId} pausing: GitLab recovery mode is active",
        gitLabUserId);

    while (_gitLabRecoveryService.IsInGitLabRecoveryMode && !ct.IsCancellationRequested)
    {
        await Task.Delay(TimeSpan.FromSeconds(5), ct);
    }

    _logger.LogInformation("Sync thread for user {UserId} resuming after recovery", gitLabUserId);
}
```

Apply the same fix to the equivalent early-return check at lines 152-159 (before the backfill phase).

### 11. `CleanupService`: DST edge case in 3am NZ time calculation

**File:** `src/be/Mergician/Services/CleanupService.cs:138-142`

The code comment says "TimeZoneInfo handles DST automatically", but it does not apply here. `targetNz` is constructed directly with `nowNz.Offset`, which is the *current* NZ offset. If `AddDays(1)` crosses a DST boundary (NZ transitions in April and September), the offset used for the target time will be wrong by one hour, and the cleanup may fire an hour early or late.

Use `TimeZoneInfo.ConvertTimeFromUtc` to let the BCL resolve the correct offset for the target date:

```csharp
// Current
var targetNz = new DateTimeOffset(nowNz.Year, nowNz.Month, nowNz.Day, 3, 0, 0, nowNz.Offset);
if (nowNz >= targetNz)
{
    targetNz = targetNz.AddDays(1);
}
var targetUtc = targetNz.ToUniversalTime();

// Fix
var todayAt3am = new DateTime(nowNz.Year, nowNz.Month, nowNz.Day, 3, 0, 0, DateTimeKind.Unspecified);
var targetUtcCandidate = TimeZoneInfo.ConvertTimeToUtc(todayAt3am, _nzTimeZone);
if (targetUtcCandidate <= nowUtc)
{
    targetUtcCandidate = TimeZoneInfo.ConvertTimeToUtc(todayAt3am.AddDays(1), _nzTimeZone);
}
var targetUtc = targetUtcCandidate;
```

The current implementation calls `.ToLower()` on both operands, allocating two new strings on every call. Use `StringComparison.OrdinalIgnoreCase` instead, which compares in-place with no allocations.

// Current
public static bool ContainsIgnoringCase(this string? str, string compareTo)
{
    return str != null && str.ToLower().Contains(compareTo.ToLower());
}

// Fix

## Bootstrap

### 14. `Program.cs` outer catch block violates the "fail fast" guideline

**File:** `src/be/Bootstrap/Program.cs:102-108`

The bootstrapper guideline states: *"If an unexpected exception happens, let it throw and abort the application. Do not add catch blocks to try and fail gracefully."* The outer `catch (Exception ex)` block logs and returns exit code 1 instead of letting the exception propagate as an unhandled crash. While the exception detail is captured by Serilog, the stack trace is not visible in the terminal output, making failures harder to diagnose quickly.

Rethrow after logging so the process exits with a non-zero code and the full exception is printed:

```csharp
catch (Exception ex)
{
    Log.Fatal(ex, "Unexpected exception during bootstrap");
    Logging.LogSeparator();
    // Re-throw so the runtime prints the full stack trace and exits non-zero
    throw;
}
```

Remove the explicit `return 0` / `return 1` exits; the presence or absence of an unhandled exception is the exit-code signal.
