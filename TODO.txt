Curently the Mergician backend records each user's "last poll time" so that new activity can be checked for since that time. Having this recorded on the backend breaks if the user has multiple tabs of the dashboard open, as this will cause some activity to be missed in some tabs. Instead, refactor to remove the last poll time from the database (and the supporting code around that), and make it so that the "last poll time" is passed in from the frontend. To account for clock skew between the frontend and backend, do not use the clock on the front end to determine the next poll time. Instead, figure out a way to have the backend return a "next poll time" in either every reponse to a request for dashboard data, or with a separate event, whichever makes the most sense. Ensure that the "next poll time" is calculated by the backend as soon as a request for new data arrives from the frontend and _before_ contacting Gilab for new data, so that we don't miss any data. For the initial dashboard load the frontend should not send a "last poll time"; in this case the backend first sends the merge groups already in the database for that user, if any, so as it does so it could see which had the newest activity time recorded against it and that can be the last poll time. If there is no records in the DB to send back, or the newest entry is more than 14 days old, then clamp the search to 14 days only.

Ensure the integration tests still work, then commit and push these changes before moving on.
