Task 1)
Get rid of the option to control if "auto-rebase" is enabled or not. If a Merge Group (MG) is set to auto-merge, then we should always rebase the branch if it is out of date. Get rid of all the code supporting the configurability of this, remove the field from the database.

Task 2)
Currently, the status of a branch (where a Merge Request exists) is derived by the Mergician backend by looking at things like approval status, if rebase is needed, conflicts, etc. Instead of calculating these ourselves, just use Gitlab's API to get the detailed MR status and from those results derive our status. Howering over the status chip should still show all the blocking reasons, but these can be derived from Gitlab respnse. We still need to show things like Approvals in the UI as we do now, just not take them into account for status.

We only care about status of a branch and it's associated MG if the MG is set to auto-merge, so don't even bother checking the MR statuses for the other MGs.

The statuses as far as we are concerned should be:
- "Waiting" if Gitlab says the only reason the MR is blocked is because it is waiting for the current pipeline to finish _and_ there are no failed build jobs in the current pipeline.
    - Note that this is also the only status that allows a MG to get added to a merge queue and to remain in the merge queue.
- "Blocked" for all other reasons.
