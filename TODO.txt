Redo how the auth token is retrieved in the backend. Instead of having a polymorphic `GitlabAccessUserBase`, create a new service called `GitlabUserFactory` that has two public methods: `GetCurrentUser` and `GetServiceUser`. Move the token fetching functionality from the child classes of `GitlabAccessUserBase` into this factory as `GetCurrentUserToken` and `GetServiceUserToken` that the public methods can use.

Then rename `GitlabAccessUserBase` to `GitlabAccessUser`, make it non-abstract and delete the child classes. The `GitlabAccessUser` must be constructed with a non-null access token and apiBaseUrl. The above public methods in the factory should each return an appropriate `GitlabAccessUser` (or null if not authenticated).

Only the Controllers in the backend should be using the `GitlabUserFactory`. They should get the appropriate user and pass it to services, which should work with `GitlabAccessUser`.

Do this change, then commit and push so we have a checkpoint of this code before moving on. I want to be able to revert to this state if the subsequent refactor does not work out.

Now see if we can make the token passing and authentication management more efficient and standard practice. Currently every method in every controller checks for the logged in user token. This is a lot of noise that will build up as we add more methods. Look at the best practices for authentication with OAuth and ASP.Net Core and figure out something that avoids all this duplication. Note that we don't _need_ to deal with the logged in user and the service user the same way. If it makes sense to deal with the OAuth logged in user with some traditional methos in ASP.Net and deal with the service user separately, then do so. The service user will only be used by the backend to perform background maintenance tasks and it only needs to fit into the pattern as far as being able to call GitlabService methods with a `GitlabAccessUser` so that the Gitlab service calls themselves don't need to know if they are being called by a logged in user or the service user.
