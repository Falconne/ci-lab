We need a database to store persistent data in Mergician. Use Dapper to provide ORM and DbUp to handle migrations. Use the recommended best practices for these libraries to structure the code and implement and abstract the DB usage well.

Add database connection settings support in a similar way to the other settings.

In the mergician docker compose script, implelemt a DB service using Postgres for internal testing, and add the necessary configuration through the .env file for the connection and credential settings. Include bootrapping of the database as part of the docker compose setup, setting up the sysadmin credentials for Mergician to connect. The DB that Mergician uses will be exclusive to it, so it is ok for it to have sysadmin credentials. With these it should try to do as much of the initialisation itself so we keep the docker compose init minimal.

The initial usage of the database will be to cache the results of the user activity we have seen, so we don't have to query Gitlab for as much data on subsequent dashboard loads.

Create several tables:
* UserActivity: to store Gitlab username and last poll timestamp, so subsequent queries by the same user can be done from this timestamp onwards (then updated). Replace the current polling system which just goes back a fixed number of seconds with this method.
* BranchInProject: store an Id (primary key), BranchName, ProjectId, ProjectName
* MergGroup: store an Id (primary key), Name (this should just be the branch name the merge group is created for by default), LastUpdateTime (update this any time fresh activity is seen for any branch in the group).
* BranchesInMergGroup: A table to store the one to many links from MergeGroup to BranchInProject rows (as per the current logic on the frontend, a MergeGroup is a collection of BranchInProject with the same branch name).
* UsersInMergeGroup: This should map Gitlab usernames to MergeGroups based on what users have had a branch in that merge group in their activity.

Add the above tables and refactor the backend code to use them. When a user loads their dashboard, return the data from the DB first (latest entries first). Ensure that only merge groups associated with the user in UsersInMergeGroup are returned. Also, ignore anything older than the queried timespan based on the LastUpdateTime of the MergeGroup. Then query for new data from the user's last access timestamp. Before new data is returned, add it to the DB first. Update the DB if the data already exists (i.e. there is new activity on an already tracked branch, we need to update the LastUpdateTime of the MergeGroup and possibly the `UsersInMergeGroup` table).

When returning data to the user, keep the current structure rather than changing to a MergGroup structure (i.e. we return one branch entry at a time). However, return the MergeGroupId from the DB for each branch row, and udpate the frontend to use this to group branches into merge groups instead of grouping by branch name. This applies to subsequent updates as well.

When returning data from the database, first check if the branch still exists and if not, delete it and its references from the DB. If a MergeGroup has no branches left, delete it and its references in UsersInMergeGroup, etc.

Make sure that during regular polling, if a branch a user's frontend is requesting updates for is gone, it is removed from the DB and the frontend is notified to remove it.

If a user has new activity in a branch that is already in a MergeGroup they are not associated with, add them to the MergeGroup in the DB and ensure this is reflected in the frontend (i.e. they should now see this MergeGroup and its branches in their dashboard).

Do not store MR or approvals info in the DB. This should be queried live.

Also, think about what we are trying to achieve, by using the database to cache activity and if you can think of any other sensible changes we should do to preserve data integrity or and report all relevant data to the user, then implement that as well.
