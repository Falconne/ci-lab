The homepage dashboard currently waits for all of a user's activity data to be collected before rendering the table. Make it so that the table starts filling in as soon as each activity line or data point is discovered, so the UI updates dynamically. This means the front end needs to be dynamic: a row for a branch and repo can appear before the MR status and approvals count is collected, but then be updated in place when extra data is read. There should be a small loading animation or placeholder in the cells that are waiting for data. Make sure to skip over events for branch-repo combos that are already discovered, we don't want duplicates in the UI. Also, if a new brach-repo combo is found where the branch matches that of an existing row in the UI, add it below the existing row, and group them the way it is currently done, with the cells for the branch name merged. This is known as a "Merge Group". So the activity table is a table of Merge Groups, even if some rows only have one branch-repo combo. In the UI these groups are currently called a `BranchGroup`. Rename this to `MergeGroup` to reflect the preferred terminology. Rename any other symbols related to this to reflect the new terminology. Also, the activity fetching in the backend is done in the ActivityController. The logic should be move to a GitlabActivityService instead.
