Task 1)
Controllers should return strong types where possible. For example, prefer returning `ActionResult<T>` over just `IActionResult`. Refactor existing controllers that could be improved this way and adjust the frontend if needed.

Task 2)
If Gitlab goes down, we must handle this gracefully in the backend. If any Gitlab API call fails in an unexpected way, throws an exception, or returns an unexpected 5XX error, then the frontend and backend should go back into the "startup" mode, but skip over the DB migrations and other checks and go straight to the Gitlab health check (if this is not the last check, then move this to be the last check). Wherever the current code catches a generic Exception from a Gitlab API call should be replaced with this functionality.

Exceptions from parsing json returned from Gitlab can be handled the same way as a bad response code.

Every Gitlab API call should retry 3 times with a linear backoff strategy, starting withe a delay of 2 seconds and ending with a delay of 10 seconds. Once the retries are exhausted, then go into the "startup" mode as described above.

Centralise the core Gitlab API call into a single method in a new service so that retries and returning to startup can be handled from a single location. Refactor the code so that the "return to startup" mode is handled in a clean way. If necessary you can use a custom exception to return far enough up the call stack.

Ensure that Gitlab calls that happen purely on the backend (e.g. in `RunUserSync` and for cleaning up old branches) aren't affected by the "return to startup mode" functionality. They can still use the retry logic and when the retries are exhausted, they can log the error and exit their current loop, as all backend routines should be running in a loop or a timer and they will get a chance to try again later.
